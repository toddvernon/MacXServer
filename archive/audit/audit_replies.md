## Pattern I noticed across this category

The recurring shape is "we ship the empty/uniform/defaulted form of the reply because the populated form needed feature work we hadn't done yet, and the corpus never punishes the difference." QueryFont's charInfos=[] + allCharsExist=true is the same shortcut family as QueryKeymap returning all-zeros, QueryExtension always saying `present=false`, GetMotionEvents always returning nEvents=0, and ListInstalledColormaps always returning the single default. Each is plausible-shaped on the wire; each will surface the moment a client probes it for the populated case (xterm with proportional font, xeyes pupil trail, any client that hard-requires SHAPE).

A secondary pattern: requests with multi-value semantics (GetProperty's type / longOffset / longLength; QueryFont's encoding range; SetPointerMapping → GetPointerMapping roundtrip) have their auxiliary fields silently ignored, not validated. The handler reads the bits the corpus sets and drops the rest. That's the "drops a flag/bit silently" pattern signature applied to reply requests rather than mutation requests — same disease.

Tertiary: cmap-argument validation is uniformly missing on QueryColors / LookupColor / GetSelectionOwner / etc. — the BadColormap/BadAtom doors aren't wired. A single-cmap, all-atoms-pre-interned corpus never trips them.
