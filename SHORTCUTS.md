# Shortcuts and stubs

Running list of places we hardcoded, stubbed, or otherwise corner-cut to make something work. Each entry should die eventually; the point of the file is to make sure none of them is forgotten.

## Convention

- **When you hardcode** something to make a bigger thing work, add an entry here in the same change.
- **When you replace** a hardcode with a real implementation, delete the entry (or move it to the Closed section at the bottom).
- Periodically review the open list. If it's growing and not shrinking, that's a signal to stop adding features and clean up.
- Each entry: file or component, what's stubbed, why, what "real" would look like.

## Open

- **SetupAccepted is hardcoded.** Lives in `Sources/SwiftXServerCore/ServerConfig.swift` as `ServerConfig.default`. One screen 1280×1024, one PseudoColor 8-bit visual (id 0x22), one pixmap format (depth=8 bpp=8 pad=32), vendor "swift-x", root=0x28, defaultColormap=0x21, resourceIdBase=0x4400000. The matched-Sun values are used so unit tests can replay the captured Sun C2S byte stream against our server. Real version: derive from actual macOS display geometry, expose both PseudoColor 8-bit and TrueColor 24-bit per `DECISIONS.md`.
- **GetProperty returns empty for unknown properties.** `Sources/SwiftXServerCore/ServerSession.swift` `case .getProperty`: serves stored props from `PropertyTable` when present, returns `GetPropertyReply.empty` otherwise. xclock's `GetProperty(RESOURCE_MANAGER)` hits the empty path and falls back to compiled-in defaults. Real version: maintain a server-side Xrm database (loaded from `xrdb`-equivalent or `~/.Xdefaults`).
- **AllocColor returns synthetic monotonic pixel values.** `Sources/SwiftXServerCore/ColorTable.swift` allocates from `nextPixel = 16` and caches `pixel → RGB` for M3's draw-time lookup. Real version: a proper PseudoColor colormap implementation with shared cells, freelist, and StoreColors.
- **QueryFont reports cell-snapped metrics for the resolved Mac substitute.** `Sources/SwiftXServerCore/ServerSession.swift` `case .queryFont` looks up the FontEntry's resolved metadata and returns a `QueryFontReply` with min/max bounds = cell width, ascent/descent from the substitute font, range 32..126, allCharsExist=true, charInfos=[] (monospace optimisation). Phase 4 polish: extend range to full Latin-1, populate charInfos for proportional fonts.
- **OpenFont parses XLFD or alias and stores the resolved Mac substitute.** `case .openFont` runs `FontResolver.resolve(name:)` and stores the resolved font on the FontEntry. The bridge instantiates the CTFont lazily on first text render. Phase 4 polish: cursor font (`cursor`) is treated as a regular font; needs separate cursor-font handling.
- **CreatePixmap depth=1 / PutImage just track and drop pixels.** `case .createPixmap` records geometry; `case .putImage` is a silent no-op. xclock's icon bitmap and mask never get used in rootless mode (the NSWindow gets its icon from the app bundle). Real version: when we care about iconified state, bridge depth=1 pixmaps to NSWindow icon.
- **Draws targeting pixmaps or the root silently drop.** `ServerSession.validateDrawTarget` (used by every poly*/image-text handler) returns nil with a log line for `isKnownDrawable=true` but `topLevelAndOffset=nil` drawables. **This is a deliberate ledgered lie**: per the XError-honesty policy, BadImplementation would be the spec-correct error, but emitting it breaks dt-apps which routinely draw into pixmaps as backing buffers before CopyArea-ing to a window. The CopyArea-from-pixmap path is also unimplemented (separate entry) and would also error in that mode. Real version: render to pixmap-backed CGBitmapContexts, then CopyArea blits pixmap→window bitmap. Until that ships, both ends of the pattern silent-drop together so clients keep working.
- **ChangeWindowAttributes honors EventMask + BackPixel + BorderPixel.** The remaining bits (Colormap, BitGravity, BackingStore, BackPixmap, etc.) are still dropped. BackingStore stays ignored permanently; the others are post-PoC.
- **GC state is parsed into `[bit: value]` on every CreateGC + ChangeGC.** `Sources/SwiftXServerCore/ResourceTables.swift` GCEntry stores `values: [UInt32: UInt32]`; ChangeGC overwrites prior values by bit. `GCState.materialise` reads from the dict. Only the bits we actually render (foreground / background / lineWidth / fillRule / font) get pulled out today; everything else still passes through the dict but isn't honoured at render time.
- **EnterNotify / LeaveNotify shipped 2026-05-09 with the standard X11 detail-field rules.** Tracking area on FlippedXView delivers `mouseMoved` / `mouseEntered` / `mouseExited`; bridge plumbs to ServerSession's `handlePointer*` methods which compute the crossing chain (Ancestor / Inferior / Nonlinear / Virtual / NonlinearVirtual) and emit only to windows whose eventMask has EnterWindowMask or LeaveWindowMask. xcalc's Athena Command buttons highlight on hover. (Active-grab crossing modes — NotifyGrab / NotifyUngrab — pending GrabPointer/GrabKeyboard.)
- **Expose on map walks the descendant tree.** `MockWindowBridge.emitMapSequence` (shared between Mock and Cocoa bridges) emits Expose on the top-level if its event mask has ExposureMask, then on each already-mapped descendant whose event mask has ExposureMask. Region Step E1 (2026-05-13) replaced the full-rect Expose with per-clipList-rect emission in window-local coords — fully-covered descendants emit zero Exposes; partially-obscured ones emit one Expose per visible rect with `count` set to the remaining siblings.
- **Synthesized ReparentNotify uses a constant fake parent ID.** `MockWindowBridge.syntheticParentId = 0xC0FFEE00`. xclock and most apps don't care what this value is, just that the event arrives. If a client tries to manipulate the synthesized parent, they'll get an unknown-window response.
- **NSWindow opens at fixed (100, 100) on screen.** `CocoaWindowBridge.mapTopLevel` ignores the X CreateWindow geometry's x/y for the screen position. Real WMs honour `WM_NORMAL_HINTS` and various positional hints. For now the X coords just decide window size.
- **Window background is white if BackPixel is unset.** `windowBackground()` falls back when CWBackPixel isn't in the CreateWindow valueList. xclock sets it explicitly, so this is rare in practice.
- **Color resolution falls back to black for unknown pixels.** `ServerSession.resolveColor` returns black when `ColorTable.rgb(for:)` is nil. Replayed captures hit this constantly because the captured CreateGC bytes reference pixels that the *original* Sun server allocated, not ours. For live clients this is fine because they see our AllocColor reply and use our pixel values.
- **XError emission partially shipped 2026-05-14.** Framer's `XError.encode(...)` and `ServerSession.emitError(...)` are wired (`Sources/Framer/ServerMessage.swift`, `Sources/SwiftXServerCore/ServerSession.swift`). The unknown-opcode dispatcher path emits `BadRequest` (covers extension opcodes from extensions we don't install plus core opcodes whose body decoder we haven't written). **Still silent-dropping or faking success**: decode failures within known opcodes (the framer throws but the session swallows without emitting BadLength/BadValue); the per-request validation pass that should emit BadWindow / BadGC / BadDrawable / BadAtom / BadPixmap / BadCursor / BadFont when a referenced resource ID isn't in our tables; cross-window CopyArea (separate SHORTCUTS entry); ConvertSelection-from-client (separate entry); PutImage on depth=1 (separate entry); RecolorCursor (separate entry); GC clip-rectangles + dashes + passive key grabs (tracked-but-not-honored stubs that should either implement or emit honest errors — separate entries). Each conversion needs replay-test impact weighed; the captured-app test suite tolerates BadRequest on the baseline's allowed extension opcodes and fails on anything else.
- **Drawing happens on main thread, but read thread doesn't wait.** The session calls bridge.draw* from the read thread; bridge does `DispatchQueue.main.async`. So drawing closures queue up. They run in the order issued because the main queue is serial. But if main is busy/blocked, drawings pile up and don't appear. Fine for xclock-paced workloads; xterm will need a different model.
- **NSWindow resize discards prior pixels.** On `windowDidEndLiveResize` we allocate a fresh CGBitmapContext at the new size, paint the top-level + every mapped descendant's bg via paintWindowRects so the bitmap matches the X bg pixel (otherwise `xterm -bg black` flashed white in the newly-uncovered area), and emit ConfigureNotify + Expose on the top-level AND on every descendant with ExposureMask. xterm relies on the descendant Expose specifically (Xt's geometry manager re-lays out and the inner DrawingArea redraws on Expose). During a live drag we keep the OLD bitmap; FlippedXView draws it at its native size pinned to top-left, and the view's layer.backgroundColor (synced to the X bg via setTopLevelWindowBackground) fills the surrounding region so the visible content stays in the right colour without stretching.
- **Multi-client routing relies on bridge handler fan-out, not strict per-window dispatch.** `CocoaWindowBridge` keeps one list of handlers per event type and fires every registered handler on each event. Each session filters by `windows.get(topLevel) != nil` so non-owners no-op. This works because session windows live in non-overlapping resource-id ranges, but it's O(N sessions) per event. Fine for 2-3 clients; if we ever run 50 clients we'll want a (windowID -> session) map for direct dispatch.
- **Keyboard map is US-ASCII only.** `Sources/SwiftXServerCore/USKeymap.swift` covers letters / digits / punctuation / arrows / modifiers via macOS NSEvent virtual keyCode +8 = X keycode. Phase 4: international layouts, dead keys, IME.
- **Cross-window CopyArea emits BadImplementation; pixmap-source/dest unimplemented.** `handleCopyArea` validates drawables (unknown drawable → BadDrawable referencing the bad ID) before checking the bridge, then emits BadImplementation when src and dst resolve to different top-levels or either is a pixmap. Same-window scrolls (xterm) still work. Real version: shared-bitmap blit between top-levels, plus actual pixmap backing once we ship that.
- **Cursor substitution table covers ~10 X glyphs**, NSCursor side. CreateGlyphCursor / FreeCursor / CWCursor in CreateWindow + ChangeWindowAttributes shipped 2026-05-09. NSCursor mapping in `CocoaWindowBridge.nsCursor(forXCursorGlyph:)` covers XC_xterm/iBeam, XC_left_ptr/arrow, XC_crosshair, hand cursors, fleur/openHand, and the resize variants. Unknown glyphs fall back to .arrow. RecolorCursor is silent-dropped (we override with NSCursor anyway). CreatePixmapCursor — fully-custom client-pixmap cursors — is also silent-dropped; if any client uses one we'd render the macOS arrow.
- **NoExpose always emitted after CopyArea.** We don't track GC `graphics-exposures`, so we send NoExpose unconditionally. Spec says only emit if graphics-exposures=True; defaults to True so most clients are fine. A client that explicitly sets it False and ignores NoExpose would see spurious events.
- **Selection roundtrip only handles STRING, not INCR.** `case .changeProperty` intercepts converted text on `selectionSinkWindow` and pushes to NSPasteboard, decoding bytes as ISO-8859-1. Big selections (xterm uses INCR > 64K) and non-STRING targets (UTF8_STRING, COMPOUND_TEXT) aren't handled. Typical xterm selections are short ASCII so this works in practice; flag if multi-MB selections start truncating.
- **ConvertSelection from clients isn't routed.** Opcode 24 still falls into `.unknown`. We *emit* SelectionRequest events for our own internal copy roundtrip, but a client issuing ConvertSelection (e.g. middle-click-paste in xterm pulling its own PRIMARY) would not get serviced — the server would log unknown opcode and the client would hang waiting for SelectionNotify. Single-client xterm doesn't trip this in practice; second client or xterm's own paste path would.
- **CLIPBOARD selection isn't mirrored.** Only PRIMARY (atom 1) is tracked for the copy roundtrip. xterm-on-u5 only writes PRIMARY so this is fine; modern xterm builds and Motif apps that write CLIPBOARD won't end up on NSPasteboard until we add it.
- **Server appears in the Dock.** `setActivationPolicy(.regular)` is what gives us the menu bar; the side effect is a Dock icon. Suppressing it needs an `Info.plist` with `LSUIElement=1` (or `NSUIElement=YES`), which Swift Package executables don't ship by default. Either start producing an .app bundle with a real Info.plist, or call the private `TransformProcessType` runtime trick. Cosmetic, deferred.
- **Listening IP detection picks the first `en*` interface.** `primaryLocalIPv4()` in `main.swift` returns en0 if up, otherwise the first up `en*` interface; skips `awdl*`, `utun*`, `bridge*`, etc. Hosts with multiple LAN interfaces (Mac Pro with both Wi-Fi and Ethernet on different subnets) may show the wrong one. Workaround today is `--host <ip>` to override.
- **Synthesized xlsfonts is the Phase 1 set, not the full ~100.** `SynthesizedFonts.phase1Names` ships ~40 entries (cell aliases + substitute families × 5 sizes × medium/bold roman iso10646-1). Phase 4 expands to italic + iso8859-1 cross product.
- **Reply-generating opcodes that still fall back.** QueryBestSize is impl. The 2026-05-10 sweep landed QueryTree, GetAtomName, TranslateCoordinates, QueryPointer, ListExtensions, QueryKeymap as proper replies. None are silent-dropped now — see OPCODE_STATUS.
- **GC clip rectangles honored for poly/image-text draws as of 2026-05-14.** `WindowBridge` draw methods accept a `clipRectangles: [Rectangle]?` parameter; `ServerSession` threads `state.clipRectangles` (origin-offset-applied, top-level-coords) from `GCState.materialise` into every poly/image-text call site. `CocoaWindowBridge.withClip` applies the union via `CGContext.clip(to:)` inside saveGState/restoreGState. nil = unbounded; empty array = clip-everything (skip the draw per spec). **Still NOT honored on CopyArea** — the same-window pixel-blit path uses direct bitmap memmove (required for correct overlap), not CGContext drawing, so clip would need a different implementation (clip the dst rect against the clip-region before iterating). xterm's scroll case doesn't set clip, so no current capture trips this; logged at the CopyArea bridge method when non-nil clip is dropped. Should close xfontsel listbox / Athena Viewport scrolling / Motif text widget bounds overflow visually once verified on u5.
- **GC dashes honored on stroke methods as of 2026-05-14.** `WindowBridge` stroke methods (drawPolySegment / drawPolyLine / drawPolyRectangle / drawPolyArc) take `dashes: [UInt8]?` + `dashOffset: UInt32`; `CocoaWindowBridge.applyDashes` calls `CGContext.setLineDash` inside the clip scope. Single-byte patterns are duplicated to [N, N] per X spec (one byte = equal on/off runs). Fills, text, and CopyArea don't take dashes — they don't stroke. Solid lines still correct for the no-SetDashes case.
- **GrabKey passive grabs honored as of 2026-05-14.** `findActivatablePassiveKeyGrab` mirrors the button-grab match algorithm (AnyKey=0 + AnyModifier=0x8000 wildcards, modifier mask in low 8 bits, grab-window must be on the natural target's ancestor chain). `handleKeyEvent` routing precedence is now: active keyboard grab → matching passive key grab → focus window → keyTarget fallback. Should make quickplot accelerator keys fire (verification pending u5 hardware).
- **WarpPointer doesn't move the macOS pointer.** Updates `lastPointerXY` so QueryPointer reflects the warp; doesn't call CGWarpMouseCursorPosition (jarring user behavior in rootless). Clients that warp-and-expect-pointer-visibly-jumped see the discrepancy.
- **SendEvent doesn't propagate.** Sets the synthetic-event high bit on byte 0 and writes to the client. Propagation up the ancestor chain (when propagate=true) is NOT honored — delivers to destination as supplied. Predefined destinations (PointerWindow=0, InputFocus=1) are not resolved.
- **DestroyWindow / UnmapWindow / ReparentWindow now emit SubstructureNotify to parents as of 2026-05-14.** New `ServerSession.notifySubstructure(parent:build:)` helper: if `parent` has SubstructureNotifyMask, builds and appends the event with `event = parent`. DestroyWindow (descendant path) emits DestroyNotify(event=parent), UnmapWindow (descendant path) emits UnmapNotify(event=parent), ReparentWindow emits ReparentNotify(event=parent) to BOTH old and new parent (skipped if they're the same window). **Still NOT emitting**: top-level destroy/unmap (root isn't in our windows table, no current client listens via root); DestroySubwindows / UnmapSubwindows (the recursive iterations); MapNotify(event=parent) on MapWindow / MapSubwindows. Those are bigger touches through the bridge layer; separate commit when needed.
- **RecolorCursor no-op.** We substitute NSCursor glyphs and don't honor X-protocol cursor fg/bg colors.
- **No SHAPE extension.** XShape is what gives xeyes its non-rectangular outline (eye shape clipped to an ellipse). Without it, xeyes renders inside its rectangular bounding window — visible as a slight rectangular "halo" around each eye where the white sclera meets the window edge. Cosmetic, low-priority. xclock has the same issue for its non-square clock face.
- **Popup menu placement is over the launching button instead of below it.** Visible in both xfontsel (Athena MenuButton + SimpleMenu) and quickplot (Motif menu). Override-redirect popup window is positioned where the client requested, but Motif/Athena typically request a position that assumes the menu drops downward FROM the bottom edge of the launching button; our placement appears to land directly on the button. Possible causes: we don't honor `_MOTIF_WM_OFFSET` on the popup, or our coordinate origin for override-redirect windows is off by the height of the launching widget. Cosmetic but consistent across toolkits. (quickplot works as of 2026-05-10 — earlier entry about callbacks not firing was fixed via the MATCH_SELECT time fix; the menu-placement issue is the visible residue.)
- **Motif Text widget character spacing.** Plot text + xcalc PolyText8 render correctly so global font metrics are fine; Motif's XmText widget specifically shows odd inter-glyph spacing. Likely Motif treats Text as column-based and we're returning Core Text actual-advance values that don't snap to its column model. Inspect when revisiting.
- **Idle 6-request poll loop in quickplot.** When quickplot sits idle, the server log streams a tight repeat of `GetWindowAttributes, GetGeometry, ChangeGC, PolySegment, ChangeGC, PolySegment` — top-shadow + bottom-shadow stroke pair on something that introspects window state each pass. Probably a Motif Text widget cursor blink or focus-ring animation timer that never decides nothing has changed. Wastes server CPU but renders correctly. Compare to gold capture before claiming it's our bug — Motif may behave this way against any X server.
- **Fake CDE customization daemon for dt-apps.** Every CDE dt-app (dtcalc, dtterm, dthelpview, dticon — verified 2026-05-10) probes a `Customize Data:N` selection at init; on a real Solaris box that selection is owned by `dtsession`, and the app reads an `SDT Pixel Set` property from the daemon's window for the system palette manifest. With no owner, dt-apps fall to a formal ConvertSelection that wedges indefinitely (Solaris-Xt's "no daemon" path is apparently untested since CDE always runs dtsession). `ServerSession.init` registers a server-internal stub window (`0xFFFE_0003`) as a child of root, claims selection ownership of `Customize Data:0`, and pre-sets the `SDT Pixel Set` property with the exact 94-byte ASCII underscore-list captured from the real u5 daemon. The ConvertSelection handler short-circuits any selection owned by a window in our internal range (`>= 0xFFFE_0000`) by writing empty bytes to the requestor's property and emitting SelectionNotify success. Replaces the previous SelectionNotify(property=None) flow that left dt-apps wedged.
- **Hardcoded CDE palette in ColorTable.** dt-apps reference colormap pixels (1, 2, 4, 5..14, etc.) directly as `CWBackPixel` based on the SDT Pixel Set manifest — those pixels are pre-allocated by the real CDE daemon with the user's chosen palette. We never allocate them, so without seeding `resolveColor()` falls back to black and dt-app windows paint entirely black. ColorTable's init seeds pixels 1..23 with a fixed approximation of the CDE "Default" colour scheme (mid-cool-grey panel bg = pixel 9, slightly bluer LCD bg = pixel 14, top shadow = pixel B light grey, bottom shadow = pixel C dark grey, text = pixel 4 near-black). Hardcoded values rather than a real palette — every session gets the same colours regardless of which CDE scheme the user picked on the Sun side. Sufficient for dt-apps to render as grey-on-grey with proper shadow accents; doesn't match colour-scheme settings on the SS5/u5.
- **Hardcoded `SDT Pixel Set` property bytes.** The 94-byte ASCII string `2_4_8_6_7_5_9_d_b_c_a_e_12_10_11_f_13_17_15_16_14_9_d_b_c_a_9_d_b_c_a_e_12_10_11_f_9_d_b_c_a_1` is captured verbatim from u5's real CDE daemon (dtcalc-sun.xtap seq=29 reply, 2026-05-10) and embedded in `ServerSession.init`. It's the palette pixel-index manifest dt-apps consume. Same string every session; doesn't reflect any user CDE colour scheme.
- **dt-Motif widget chrome doesn't redraw on Expose — Expose volume now matches gold; live verification pending.** dt-apps render their containers + LCD area + any window with explicit non-bg `BackPixel`, but the deep button hierarchy historically rendered as flat panels with no visible button shadows or labels. Per gold-vs-swiftx trace diff 2026-05-10: gold emits ~7 Expose events for the whole calculator boot, we emitted 451. Region Steps E0+E1+E1.5+E2 (2026-05-13) collapsed our dtcalc Expose count from 248 (post-Step-B) → 144 (E1) → 8 (E1.5+E2), matching gold within 1. **What this fixes**: the "Motif PushButton ignores spurious Expose flood" hypothesis is now ruled out — our stream is no longer a flood. **What's still open**: whether Motif actually renders the chrome under the new sparse stream needs u5 hardware to verify. If chrome still doesn't appear, the remaining likely culprits are: (a) VisibilityNotify emission (we don't emit it yet); (b) a SubstructureNotify variant of MapNotify; (c) some Motif gate we still haven't identified. Gate this entry on the first live-u5 dtcalc boot post-2026-05-13.
- **Cross-NSWindow drag tracking via NSEvent.addLocalMonitorForEvents.** XQuartz uses macOS-private `xp_*` kernel APIs (`xp_lookup_native_window`, `xp_window_id`) to inject pointer events at a layer below NSEvent — see `reference/xquartz-xserver/hw/xquartz/X11Application.m:246-480`. We don't have those APIs from a regular Swift AppKit app. Workaround: when an X-protocol pointer grab activates (passive grab, implicit grab on first ButtonPress, or explicit GrabPointer), `CocoaWindowBridge.startCrossWindowDragTracking` installs a local NSEvent monitor for `[.*MouseDragged, .*MouseUp]` so we can intercept events before AppKit's responder chain. The monitor computes the global pointer position, looks up which managed NSWindow contains it (popup-level NSPanels first per z-order), translates to that window's logical X coords, and routes via `fireMouseDragged` / `fireMouse`. The monitor consumes the event so the origin's FlippedXView doesn't double-fire. Released when the grab ends. This is necessary because AppKit binds `mouseDragged` to the view where `mouseDown` originated; without it, drag-tracking through Athena/Motif menus (origin in main NSWindow, target in popup NSPanel) silently fails. Caveat: AppKit's `event.deltaX/deltaY` is unreliable after Spaces/Fast-User-Switching per X11Application.m:929-936; we use `locationInWindow + window.convertPoint(toScreen:)` which sidesteps that issue.

## Closed

- **Descendant geometry change leaves stale pixels on parent.** Fixed via Region Step E1.5 (2026-05-13). `ConfigureWindow` on a non-top-level captures the pre-move `borderClip`, computes `oldBorderClip - newBorderClip` after the recompute, and via `ServerSession.repaintParentOverUncovered` paints the parent's bg over the delta (top-level coords) and emits Expose to parent for that region (parent-local coords) if it has ExposureMask. xcalc Athena re-layout after resize should no longer ghost buttons at their pre-resize positions. Unverified visually until u5 live test.
