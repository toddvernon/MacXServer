# Shortcuts and stubs

Per the XError-honesty policy locked into `CLAUDE.md` (Working
conventions) and `DECISIONS.md` (2026-05-13 entry), this file is the
**active ledger of currently-justified lies and partial implementations**,
not a wish list. Every entry should have:

- A specific file or component reference
- What's stubbed / hardcoded
- Why it's load-bearing (or what client behavior is at stake)
- What "real" would look like

When an entry is converted to real behavior, move it to **Closed** with
a date and rationale.

## Convention

- **When you hardcode** something to make a bigger thing work, add an
  entry here in the same change.
- **When you replace** a hardcode with a real implementation, move the
  entry to the Closed section with a date.
- Periodically review the open list. If it's growing and not shrinking,
  that's a signal to stop adding features and clean up.

## Open

### Resources, tables, replies

- ~~**SetupAccepted is hardcoded.**~~ Partially closed 2026-05-27 (audit). `ServerConfig.init(displayConfig:)` now derives screen dimensions, mm, and scale from `NSScreen.main` via `DisplayConfig.forMainDisplay()` at startup. Still open: only PseudoColor 8-bit visual advertised (no TrueColor 24-bit per `DECISIONS.md`); matched-Sun resource-id-base and colormap IDs still hardcoded for replay-test compatibility.
- **DisplayConfig picker silently falls back to scale=1 for sub-1080p non-Retina displays.** `Sources/SwiftXServerCore/DisplayConfig.swift:101-108` ships a "scale=1 at native pixel dimensions" fallback when no preset (1280×900 / 1280×720 / 1152×720 / 1008×648 / 960×540, at scales 3 then 2) fits the display's native pixel dimensions. In practice this kicks in on any non-Retina display smaller than 1920×1080 (1440×900 MacBook Air, 1366×768 laptops, etc.) since the smallest preset combo needs 960×540 @ 2x = 1920×1080 device pixels. The architectural intent in `SERVER_RESOLUTION_SCALING_AND_FONTS.md` ("Quality bar") explicitly rejects 1x rendering as "feels like XQuartz" and we don't ship that — but the fallback silently violates that contract for users on small non-Retina displays. **Real version (when it matters):** add a smaller preset (e.g. 720×450 @ 2x = 1440×900 for the MacBook Air case), or refuse to start with an actionable error message for displays below a documented minimum. Latent today because no one in the user pool is running this on a sub-1080p non-Retina Mac; surfaced in the 2026-05-31 discussion of how scaling settings should work.
- **RESOURCE_MANAGER is a 7-line hand-curated Tier 1 fixture.** `Sources/SwiftXServerCore/DefaultMotifResources.swift` publishes a small `*XmText.fontList` / `*XmLabel.fontList` / etc. set targeting `-adobe-helvetica-*` and `-adobe-courier-*` XLFDs (per MOTIF_TEXT_QUALITY.md Tier 1). Steers Motif to pick fonts that route cleanly through `FontResolver`'s substitution table to Mac fonts that render nicely at retina. This is NOT a lie on the wire; documented here so the staged delivery (Tier 2 config file, Tier 3 settings UI) has a home. Real version: full xrdb-equivalent that loads and merges user/system Xresources, exposed editably to the user.
- ~~**Root-window properties are per-session, not server-global.**~~ Closed 2026-05-27. Root-window properties now live on `ServerCoordinator.rootProperties` (server-global, NSLock-guarded). ChangeProperty/GetProperty/DeleteProperty on root route to the coordinator. PropertyNotify on root fans out to all sessions that have PropertyChangeMask on root via `RootPropertyObserver` protocol. Session-init properties (_MOTIF_DRAG_WINDOW, _MOTIF_WM_INFO, RESOURCE_MANAGER) write to the coordinator. Unblocks Motif clipboard cross-session copy/paste (CutPaste.c stores all clipboard state as root-window properties).
- **AllocColor has no freelist and no cell cap.** `Sources/SwiftXServerCore/ColorTable.swift` allocates from `nextPixel = 16` and grows the dict unbounded. Long-running sessions that churn colors via xterm `-bg foo -fg bar` cycling will leak entries. Real version: FreeColors round-trips, AllocColorCells (writable cells), StoreColors, a 256-cell cap per the PseudoColor depth-8 visual we advertise. *(Was a bigger entry before 2026-05-19 — that fix moved ColorTable to ServerCoordinator (server-global) and added shared-cell RGB matching so AllocColor(white) returns whitePixel like a real X server does. That unblocked Motif's BlackWhite-detection in dtcalc.)*
- **Color resolution falls back to black for unknown pixels.** `ServerSession.resolveColor` returns black when `ColorTable.rgb(for:)` is nil. Replayed captures hit this constantly because the captured CreateGC bytes reference pixels that the *original* Sun server allocated, not ours. For live clients this is fine because they see our AllocColor reply and use our pixel values.
- **Synthesized xlsfonts is the Phase 1 set, not the full ~100.** `SynthesizedFonts.phase1Names` ships ~40 entries (cell aliases + substitute families × 5 sizes × medium/bold roman iso10646-1). Phase 4 expands to italic + iso8859-1 cross product.

### XError emission (partial)

- **XError emission incomplete on a handful of paths.** Framer's `XError.encode(...)` + `ServerSession.emitError(...)` infrastructure (2026-05-14) and ~55 handlers have been converted across the validation sweep. The 2026-05-22 error-path sweep (`ErrorPathSweepTests.swift`, 69 tests on top of the existing 49) caught and fixed 6 silent-drop bugs across ReparentWindow.parent, WarpPointer.srcWindow/dstWindow, GrabButton.confineTo, AllocNamedColor/LookupColor/QueryColors cmap, and SetSelectionOwner/GetSelectionOwner/ConvertSelection selection-atom. Still silent-dropping or faking: (a) decode failures within known opcodes — the framer throws but ServerSession's `tryRequest` swallows without emitting BadLength / BadValue; (b) RecolorCursor pixel-values not honored (cursor itself validates → BadCursor); (c) PutImage XYPixmap + ZPixmap-at-uncommon-depths silent-dropped — Bitmap and ZPixmap depth=1/8 paths land per the Closed entries below; drawable + GC validated → BadDrawable / BadGC for all formats; (d) WarpPointer doesn't move the macOS pointer (separate entry below); (e) SendEvent propagation not honored (separate entry below). Captured-app replay tests tolerate BadAtom / BadWindow on property opcodes (Sun-server-internal IDs we never see CreateWindow / InternAtom for) and BadRequest on baseline-allowed extension opcodes.
- **ChangeActivePointerGrab doesn't validate the cursor argument.** When no active pointer grab exists the handler logs and skips entirely (current fixture behavior); when an active grab exists it stores `r.cursor` on the grab and looks up `cursors.glyph(r.cursor)` — which returns nil silently rather than emitting BadCursor. Real fix: `if r.cursor != 0 { guard cursors.glyph(r.cursor) != nil else { emitError(.cursor, …); break } }` at the top of the handler. Latent — no client we host today passes a bogus cursor here. Added 2026-05-22 by the error-path sweep.

### Drawing + pixmaps

- **PutImage XYPixmap still silent-dropped; ZPixmap at depths other than 1 and 8 also silent-dropped.** ZPixmap depth=1 and depth=8 landed 2026-05-31 (see Closed). XYPixmap needs a per-plane bitmap decoder that walks one plane per set bit in planeMask; no hosted client today exercises it. Other ZPixmap depths (16, 24, 32) are only meaningful against TrueColor visuals — we advertise depth-8 PseudoColor only, so a well-behaved client shouldn't send them. Real version when needed: per-plane XYPixmap decoder; ZPixmap-depth-24/32 paired with a TrueColor visual advertisement.
- **CopyArea: same-window memmove fast path doesn't honor GC clip.** All 5 spec-supported variants ship as of Stage 2 (2026-05-17): window→window same-NSWindow uses bitmap memmove (xterm scroll fast path); window→window cross-NSWindow, pixmap→window, window→pixmap, and pixmap→pixmap all snapshot src as CGImage, crop in image-pixel coords, blit via `CGContext.draw(image:in:)` into dst via `withDrawContext` — clip applied for free on those paths. Only the same-window memmove can't easily scissor per row without becoming much slower; xterm doesn't set clip there, but we log when a client does so future divergence is visible. Defensive depth-match BadMatch not implemented yet (window depth=0 = CopyFromParent sentinel needs resolution before comparing against pixmap depths). Motif and Athena always create pixmaps at root-depth so depth mismatches don't fire in practice for the dt-app / xterm flow we host.
- ~~**NoExpose always emitted after CopyArea.**~~ Closed 2026-05-15. `GCState.graphicsExposures` now gates NoExpose emission in both CopyArea and CopyPlane handlers. Athena ScrollBar's internal GC sets graphicsExposures=False and no longer gets spurious events.
- **GetImage reverse-maps ARGB → X 8-bit pixel via ColorTable; AA edges and unallocated colors return 0.** swift-x advertises a single 8-bit PseudoColor visual but renders to 32-bit ARGB Mac backing stores. GetImage takes each backing pixel and looks up which AllocColor-assigned pixel value maps to that exact RGB; anti-aliased glyph edges and intermediate colors that were never allocated through the colormap path return pixel 0 (white). Fine for x11perf (which uses GetImage as a sync barrier and discards the bytes). Real screenshot clients (xwd, xmag) would see text edges fade to white. Real version: either (a) match each ARGB to its nearest allocated colormap cell (CIE76 or similar), or (b) upgrade the advertised visual to TrueColor 24/32-bit so GetImage returns the actual rendered ARGB. Option (b) is cleaner architecturally but requires re-validating every client we host today against the visual-class change.

### Drawing clipping (partial — 2026-05-20)

- **GC `subWindowMode` not tracked; effectively ClipByChildren for every GC.** Per X.org `mi/migc.c:miComputeCompositeClip`, `subWindowMode=IncludeInferiors` swaps the window clip from `pWin->clipList` to `NotClippedByChildren(pWin)` so the GC draws through descendant windows. We don't read bit 0x80000 (`GCSubwindowMode`) from CreateGC/ChangeGC's value list and don't store it in `GCState`. Today's host clients (Motif, Athena, xterm) use the default `ClipByChildren` so this is latent. Real fix: ~20 lines (storage + lookup + an alternate "bare window region" computed from window bounds minus mapped descendants' borderClip). Document when a screen-capture / panner client surfaces it.
- **`paintRectsForWindow`'s OUTER (border) rect is not clipped to `borderClip`.** The inner bg rect is clipped to `clipList` as of 2026-05-20, but the outer rect (border ring) still paints unclipped over `(w + 2*bw, h + 2*bw)`. Almost every Motif widget we see has `borderWidth=0` so this never fires. Real fix: clip the outer rect to the window's `borderClip` (so it only paints the visible border ring, not over descendants). Add when a client with bordered top-levels (xeyes if its outer window has a border, classic twm-decorated apps) surfaces it.

### Window lifecycle

- **Synthesized ReparentNotify uses a constant fake parent ID.** `MockWindowBridge.syntheticParentId = 0xC0FFEE00`. xclock and most apps don't care what this value is, just that the event arrives. If a client tries to manipulate the synthesized parent, they'll get an unknown-window response.
- ~~**NSWindow opens at fixed (100, 100) on screen.**~~ Closed 2026-05-27. Cascade-placement now positions each top-level at a unique screen location on first map (shipped with the 2026-05-20 WM-emulation fix). Verified visually.
- **CWBackPixmap / CWBorderPixmap silently dropped on CreateWindow + ChangeWindowAttributes.** Bits 0 and 2 of the CW value mask are never consulted. The 2026-05-15 ChangeWindowAttributes sweep added storage for the other 13 bits but missed the two pixmap-source ones. Real version: store as a `BackPixmapKind` enum (`.none`, `.parentRelative`, `.pixmap(id)`); the `.none` and `.parentRelative` paths unblock transparent-overlay-widget and twm-style root-tiling clients without needing the pixmap-tile case (which is gated on broader pixmap pixel-storage exercise). Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` finding #5).
- **CWBorderWidth dropped in ConfigureWindow.** `case .configureWindow` reads x/y/width/height/stackMode/sibling but not borderWidth (`WindowTable.resize` has no borderWidth parameter). ConfigureNotify reports the stale borderWidth. Motif's BulletinBoard widget changes borderWidth on focus; xfontsel's info-pane uses borderWidth changes. Real version: add to the ValueListReader sweep, plumb through resize, repaint border ring on change, use the new value in the emitted ConfigureNotify. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` finding #6).
- **ReparentWindow on a mapped window doesn't emit the Unmap/Map pair.** Spec 10.4 mandates that ReparentWindow on a mapped window does an implicit UnmapWindow, parent change, then implicit MapWindow if previously mapped — emitting UnmapNotify and MapNotify around the ReparentNotify. Our handler emits only ReparentNotify and never toggles `entry.mapped` or calls bridge unmap/map. Any toolkit that re-parents a mapped widget mid-session breaks; virtual-root WMs break. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` finding #8).

### Input + cursors

- **WarpPointer doesn't move the macOS pointer.** Updates `lastPointerXY` so QueryPointer reflects the warp; doesn't call CGWarpMouseCursorPosition (jarring user behavior in rootless). Clients that warp-and-expect-pointer-visibly-jumped see the discrepancy.
- **SendEvent doesn't propagate.** Sets the synthetic-event high bit on byte 0 and writes to the client. Propagation up the ancestor chain (when propagate=true) is NOT honored — delivers to destination as supplied. Predefined destinations (PointerWindow=0, InputFocus=1) are not resolved.
- **Cursor substitution covers 22 X glyph values mapped to ~8 NSCursors**, NSCursor side. CreateGlyphCursor / FreeCursor / CWCursor in CreateWindow + ChangeWindowAttributes shipped 2026-05-09. NSCursor mapping in `CocoaWindowBridge.nsCursor(forXCursorGlyph:)` covers XC_xterm/iBeam, XC_left_ptr/arrow, XC_crosshair, hand cursors, fleur/openHand, resize variants, and operationNotAllowed. Unknown glyphs fall back to `.arrow`. **RecolorCursor**: validates cursor → BadCursor on unknown; we don't honor fg/bg colors (NSCursor substitution). **CreateCursor (op 93)**: shipped 2026-05-18 as a stub -- pixmap source/mask are validated for existence (BadPixmap) but the bitmap contents are never decoded; cursor renders as NSCursor.arrow at crossing time via the sourceGlyph=0xFFFF sentinel. Surfaced when dtterm's Motif Help menu crashed with BadRequest on opcode 93. Real version: decode the 1-bit source bitmap, apply the mask, build an `NSCursor(image:hotSpot:)`. Custom cursor rendering for Motif resize/busy/menu indicators is the user-visible regression.
- **Keyboard map is US-ASCII only.** `Sources/SwiftXServerCore/USKeymap.swift` covers letters / digits / punctuation / arrows / modifiers via macOS NSEvent virtual keyCode +8 = X keycode. Phase 4: international layouts, dead keys, IME.
- **GetPointerMapping reports [1,2,3]; SetPointerMapping is silently dropped.** macOS scroll-wheel events translate to X buttons 4/5 elsewhere in our event path, but clients that probe `XGetPointerMapping` to decide whether to handle button 4/5 (xterm `-mc on` mode, dt's wheel-scroll, modern plotting clients) skip wheel handling because the map says "3-button mouse." Also: a left-handed user can't swap LMB/RMB because SetPointerMapping doesn't persist. Real version: report [1..5] and persist a mutable per-session mapping. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_replies.md` finding #10).
- **Cross-NSWindow drag tracking via NSEvent.addLocalMonitorForEvents.** XQuartz uses macOS-private `xp_*` kernel APIs (`xp_lookup_native_window`, `xp_window_id`) to inject pointer events at a layer below NSEvent — see `reference/xquartz-xserver/hw/xquartz/X11Application.m:246-480`. We don't have those APIs from a regular Swift AppKit app. Workaround: when an X-protocol pointer grab activates (passive grab, implicit grab on first ButtonPress, or explicit GrabPointer), `CocoaWindowBridge.startCrossWindowDragTracking` installs a local NSEvent monitor for `[.*MouseDragged, .*MouseUp]` so we can intercept events before AppKit's responder chain. The monitor computes the global pointer position, looks up which managed NSWindow contains it (popup-level NSPanels first per z-order), translates to that window's logical X coords, and routes via `fireMouseDragged` / `fireMouse`. The monitor consumes the event so the origin's FlippedXView doesn't double-fire. Released when the grab ends. Caveat: AppKit's `event.deltaX/deltaY` is unreliable after Spaces/Fast-User-Switching per X11Application.m:929-936; we use `locationInWindow + window.convertPoint(toScreen:)` which sidesteps that issue.

### Selection / clipboard

- **Selection roundtrip only handles STRING, not INCR.** `case .changeProperty` intercepts converted text on `selectionSinkWindow` and pushes to NSPasteboard, decoding bytes as ISO-8859-1. Big selections (xterm uses INCR > 64K) and non-STRING targets (UTF8_STRING, COMPOUND_TEXT) aren't handled. Typical xterm selections are short ASCII so this works in practice; flag if multi-MB selections start truncating.
- **CLIPBOARD selection isn't mirrored.** Only PRIMARY (atom 1) is tracked for the copy roundtrip. xterm-on-u5 only writes PRIMARY so this is fine; modern xterm builds and Motif apps that write CLIPBOARD won't end up on NSPasteboard until we add it.
- **SetSelectionOwner has no time-comparison gate.** Spec 4.2.1: silently ignore when `r.time < prior.time` or `r.time > serverTime`. `ServerCoordinator.swapSelectionOwner` unconditionally overwrites without consulting the prior owner's stored time. Latent because today's hosted clients don't contend; the day two clients race for PRIMARY, a late-arrival ConvertSelection can clobber a newer owner. ~5 line fix. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_replies.md` finding #7).
- **GetProperty doesn't honor the type filter.** Spec: when the request specifies a non-zero type and the stored property's type differs, the reply returns the stored property's actual type + bytes-after=full-length + an empty value (don't transfer bytes). We never compare stored type against requested type and always transfer. This is the bug-shape the moment anyone implements INCR — the INCR pattern reads with delete=True on every intermediate chunk and gates on type match. Also: GetProperty(delete=True) currently deletes regardless of `bytes-after`; spec says delete only when bytes-after==0. Surfaced by comparison fork 2026-05-14 (`archive/comparison/risk_selections_properties.md`).

### Extensions

- **BIG-REQUESTS not advertised.** DECISIONS 2026-05-05 committed to ship; deliberately re-deferred 2026-05-14 after investigating cost. R6's server-side dispatch handler IS small (~24 lines per `Xext/bigreq.c`), but R6's decoders read body lengths from a `client->req_len` field in a context struct, not from header bytes. swift-x's `Request.decode` and every per-opcode decoder reads `lenIn4` from `bytes[2..3]`, so supporting extended-length encodings (length=0 → 32-bit extension field at bytes[4..7]) requires either (a) threading `extendedLength: UInt32?` through `Request.decode` + per-decoder use in ~5 decoders (PolyLine, PolyArc, ChangeProperty, SetClipRectangles, FillPoly), or (b) a request-rewrite trick that caps support at 65535 4-byte units (defeats the purpose). Real implementation is ~200 lines, not 24. The "wedge risk" the comparison study mentioned (client sending length=0) is now closed by the bogus-length-actually-closes fix from 2026-05-14 — we emit BadLength + close, which is spec-correct for an unsupported extension. Revisit when a client we host actually needs BIG-REQUESTS (none today). Confirmed via comparison study: only SHAPE and BIG-REQUESTS matter for the R6/CDE/Motif client mix. SHAPE shipped 2026-05-28; BIG-REQUESTS remains deferred and isn't on a critical path today.
- **SolarisIA extension missing.** Real Sun's Xsun advertises it (`QueryExtension(SolarisIA) → present=true major=135`); we report present=false. SolarisIA is the Solaris-internal Inter-client Atoms cache. When absent, Motif falls back to interning ~50 atoms one-by-one at startup. No visual bug — just latency and log noise during dt-app boot. Real fix: minimal extension implementation that batches atom interning (~half day). Surfaced in the dtcalc-sun vs dtcalc-swiftx diff 2026-05-17 (preserved findings from the archived FOLLOWUPS doc, item #3).
- **RotateProperties (op 114) not wired.** No framer decoder; falls through to BadRequest. Used by Xt for cycling icon-name properties and by some twm-era utilities. ### Multi-client gaps

- **KillClient / SetCloseDownMode accepted as no-ops.** Decoders shipped 2026-05-14 so these stop falling through to BadRequest (Xt scatters them around session lifecycle). Real implementation would route KillClient.resource → owning client → close socket, and SetCloseDownMode would persist the mode on the session for cleanup-time dispatch. No client we host today actually uses RetainPermanent / RetainTemporary close-down.
- **GrabServer / UngrabServer accepted as no-ops.** Honoring would require coordinator-mediated pause of other sessions' request dispatch. **Product choice**: swift-x is the WM; we don't host an external WM that would need GrabServer atomicity for reparenting or restacking. The apps we host (xterm, Motif, dt-apps) never call GrabServer for anything besides ttsession's property-publish handshake, which we've obsoleted. See `OPCODE_STATUS.md` row 36.
- **Cross-session AlreadyGrabbed not detected on GrabPointer / GrabKeyboard.** `pointerGrab` / keyboardGrab are session-local; we don't coordinate via ServerCoordinator. **Product choice**: same WM-emulation framing -- external WMs are the primary consumer of cross-session grab arbitration. Our rootless model handles pointer routing via NSEvent monitors and Mac-native focus, so two apps grabbing simultaneously doesn't arise in practice.
- **Substructure-redirect events (MapRequest / ConfigureRequest / CirculateRequest) never emitted.** Anywhere the spec says "if parent has SubstructureRedirectMask, redirect to the manager," we apply the request unconditionally. **Product choice**: swift-x IS the window manager. SubstructureRedirect exists so an external WM can intercept and decorate MapWindow/ConfigureWindow before they happen. We do that decoration ourselves (rootless NSWindow, cascade placement, ICCCM synthetic ConfigureNotify). Hosting an external Sun-side WM (twm, mwm) is explicitly not the primary mode -- users who want that aesthetic run mwm on the Sun and we render its decoration windows as ordinary X windows. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` pattern D).

- **Pixmap image-writer y-orientation inconsistency (scrollbar thumb vs button-bar bitmap).** Closed 2026-05-27. `PixelBuffer`'s CGBitmapContext carries a y-flipped CTM (`translateBy(0, h); scaleBy(1,-1)`) so X-protocol y-down coords pass through to fill/stroke ops. - **ColorTable moved to server-global; AllocColor honors shared cells.** Closed 2026-05-19.
