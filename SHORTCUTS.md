# Shortcuts and stubs

Per the XError-honesty policy locked into `CLAUDE.md` (Working
conventions) and `DECISIONS.md` (2026-05-13 entry), this file is the
**active ledger of currently-justified lies and partial implementations**,
not a wish list. Every entry should have:

- A specific file or component reference
- What's stubbed / hardcoded
- Why it's load-bearing (or what client behavior is at stake)
- What "real" would look like

When an entry is converted to real behavior, move it to **Closed** with
a date and rationale.

## Convention

- **When you hardcode** something to make a bigger thing work, add an
  entry here in the same change.
- **When you replace** a hardcode with a real implementation, move the
  entry to the Closed section with a date.
- Periodically review the open list. If it's growing and not shrinking,
  that's a signal to stop adding features and clean up.

### Resources, tables, replies

- **SetupAccepted is hardcoded.** Lives in `Sources/SwiftXServerCore/ServerConfig.swift` as `ServerConfig.default`. ### Drawing + pixmaps

- **PutImage validates args but doesn't decode pixel data.** `case .putImage` validates drawable + GC at entry, then silent-drops the pixel data. xeyes/xclock/dtcalc-icon all use it for 1-bit-or-8-bit icon bitmaps; without decode those glyphs don't show. Real version: depth/format-aware decoder (Bitmap d=1, ZPixmap d=1/8 cover all observed traffic per the post-Stage-0 captures) writing into the pixmap's PixelBuffer.context. The draws-into-pixmap infrastructure landed in Stage 1b (2026-05-17) â€” PutImage is what remains. Targets Stage 3 of PLAN_PIXMAP_AND_COPYAREA.
- **CopyArea: same-window memmove fast path doesn't honor GC clip.** All 5 spec-supported variants ship as of Stage 2 (2026-05-17): windowâ†’window same-NSWindow uses bitmap memmove (xterm scroll fast path); windowâ†’window cross-NSWindow, pixmapâ†’window, windowâ†’pixmap, and pixmapâ†’pixmap all snapshot src as CGImage, crop in image-pixel coords, blit via `CGContext.draw(image:in:)` into dst via `withDrawContext` â€” clip applied for free on those paths. Only the same-window memmove can't easily scissor per row without becoming much slower; xterm doesn't set clip there, but we log when a client does so future divergence is visible. Defensive depth-match BadMatch not implemented yet (window depth=0 = CopyFromParent sentinel needs resolution before comparing against pixmap depths). Motif and Athena always create pixmaps at root-depth so depth mismatches don't fire in practice for the dt-app / xterm flow we host.
- **NoExpose always emitted after CopyArea.** We don't track GC `graphics-exposures`, so we send NoExpose unconditionally. Spec says only emit if graphics-exposures=True; defaults to True so most clients are fine. A client that explicitly sets it False and ignores NoExpose would see spurious events.

### Window lifecycle

- **Synthesized ReparentNotify uses a constant fake parent ID.** `MockWindowBridge.syntheticParentId = 0xC0FFEE00`. xclock and most apps don't care what this value is, just that the event arrives. If a client tries to manipulate the synthesized parent, they'll get an unknown-window response.
- **NSWindow opens at fixed (100, 100) on screen.** `CocoaWindowBridge.mapTopLevel` ignores the X CreateWindow geometry's x/y for the screen position. Real WMs honour `WM_NORMAL_HINTS` and various positional hints. For now the X coords just decide window size.

### Input + cursors

- **WarpPointer doesn't move the macOS pointer.** Updates `lastPointerXY` so QueryPointer reflects the warp; doesn't call CGWarpMouseCursorPosition (jarring user behavior in rootless). Clients that warp-and-expect-pointer-visibly-jumped see the discrepancy.
- **SendEvent doesn't propagate.** Sets the synthetic-event high bit on byte 0 and writes to the client. Propagation up the ancestor chain (when propagate=true) is NOT honored â€” delivers to destination as supplied. Predefined destinations (PointerWindow=0, InputFocus=1) are not resolved.
- **Cursor substitution covers ~10 X glyphs**, NSCursor side. CreateGlyphCursor / FreeCursor / CWCursor in CreateWindow + ChangeWindowAttributes shipped 2026-05-09. NSCursor mapping in `CocoaWindowBridge.nsCursor(forXCursorGlyph:)` covers XC_xterm/iBeam, XC_left_ptr/arrow, XC_crosshair, hand cursors, fleur/openHand, and the resize variants. Unknown glyphs fall back to `.arrow`. **RecolorCursor**: validates cursor â†’ BadCursor on unknown; we don't honor fg/bg colors (NSCursor substitution). **CreatePixmapCursor**: silent-dropped â€” fully-custom client-pixmap cursors render as macOS arrow.
- **Keyboard map is US-ASCII only.** `Sources/SwiftXServerCore/USKeymap.swift` covers letters / digits / punctuation / arrows / modifiers via macOS NSEvent virtual keyCode +8 = X keycode. Phase 4: international layouts, dead keys, IME.
- **Cross-NSWindow drag tracking via NSEvent.addLocalMonitorForEvents.** XQuartz uses macOS-private `xp_*` kernel APIs (`xp_lookup_native_window`, `xp_window_id`) to inject pointer events at a layer below NSEvent â€” see `reference/xquartz-xserver/hw/xquartz/X11Application.m:246-480`. We don't have those APIs from a regular Swift AppKit app. Workaround: when an X-protocol pointer grab activates (passive grab, implicit grab on first ButtonPress, or explicit GrabPointer), `CocoaWindowBridge.startCrossWindowDragTracking` installs a local NSEvent monitor for `[.*MouseDragged, .*MouseUp]` so we can intercept events before AppKit's responder chain. The monitor computes the global pointer position, looks up which managed NSWindow contains it (popup-level NSPanels first per z-order), translates to that window's logical X coords, and routes via `fireMouseDragged` / `fireMouse`. The monitor consumes the event so the origin's FlippedXView doesn't double-fire. Released when the grab ends. Caveat: AppKit's `event.deltaX/deltaY` is unreliable after Spaces/Fast-User-Switching per X11Application.m:929-936; we use `locationInWindow + window.convertPoint(toScreen:)` which sidesteps that issue.

### Selection / clipboard

- **Selection roundtrip only handles STRING, not INCR.** `case .changeProperty` intercepts converted text on `selectionSinkWindow` and pushes to NSPasteboard, decoding bytes as ISO-8859-1. Big selections (xterm uses INCR > 64K) and non-STRING targets (UTF8_STRING, COMPOUND_TEXT) aren't handled. Typical xterm selections are short ASCII so this works in practice; flag if multi-MB selections start truncating.
- **CLIPBOARD selection isn't mirrored.** Only PRIMARY (atom 1) is tracked for the copy roundtrip. xterm-on-u5 only writes PRIMARY so this is fine; modern xterm builds and Motif apps that write CLIPBOARD won't end up on NSPasteboard until we add it.

### CDE / dt-app daemon impersonation

- **Fake CDE customization daemon for dt-apps.** Every CDE dt-app (dtcalc, dtterm, dthelpview, dticon â€” verified 2026-05-10) probes a `Customize Data:N` selection at init; on a real Solaris box that selection is owned by `dtsession`, and the app reads an `SDT Pixel Set` property from the daemon's window for the system palette manifest. With no owner, dt-apps fall to a formal ConvertSelection that wedges indefinitely (Solaris-Xt's "no daemon" path is apparently untested since CDE always runs dtsession). `SelectionMediator.installCDECustomizationDaemonImpersonation` (2026-05-14 refactor; was inline in `ServerSession.init`) registers a server-internal stub window (`0xFFFE_0003`) as a child of root, claims selection ownership of `Customize Data:0`, and pre-sets the `SDT Pixel Set` property. ConvertSelection short-circuits any selection owned by a stub window (id â‰¥ `SelectionMediator.stubWindowFloor` = 0xFFFE_0000) by writing empty bytes to the requestor's property and emitting SelectionNotify success.
- **Hardcoded CDE palette in ColorTable.** dt-apps reference colormap pixels (1, 2, 4, 5..14, etc.) directly as `CWBackPixel` based on the SDT Pixel Set manifest â€” those pixels are pre-allocated by the real CDE daemon with the user's chosen palette. We never allocate them, so without seeding `resolveColor()` falls back to black and dt-app windows paint entirely black. ColorTable's init seeds pixels 1..23 with a fixed approximation of the CDE "Default" colour scheme (mid-cool-grey panel bg = pixel 9, slightly bluer LCD bg = pixel 14, top shadow = pixel B light grey, bottom shadow = pixel C dark grey, text = pixel 4 near-black). Hardcoded values rather than a real palette â€” every session gets the same colours regardless of which CDE scheme the user picked on the Sun side. Sufficient for dt-apps to render as grey-on-grey with proper shadow accents; doesn't match colour-scheme settings on the SS5/u5.
- **Hardcoded `SDT Pixel Set` property bytes.** The 94-byte ASCII string `2_4_8_6_7_5_9_d_b_c_a_e_12_10_11_f_13_17_15_16_14_9_d_b_c_a_9_d_b_c_a_e_12_10_11_f_9_d_b_c_a_1` is captured verbatim from u5's real CDE daemon (dtcalc-sun.xtap seq=29 reply, 2026-05-10) and embedded in `SelectionMediator.installCDECustomizationDaemonImpersonation`. It's the palette pixel-index manifest dt-apps consume. Same string every session; doesn't reflect any user CDE colour scheme.

### Extensions

- **BIG-REQUESTS not advertised.** DECISIONS 2026-05-05 committed to ship; deliberately re-deferred 2026-05-14 after investigating cost. R6's server-side dispatch handler IS small (~24 lines per `Xext/bigreq.c`), but R6's decoders read body lengths from a `client->req_len` field in a context struct, not from header bytes. swift-x's `Request.decode` and every per-opcode decoder reads `lenIn4` from `bytes[2..3]`, so supporting extended-length encodings (length=0 â†’ 32-bit extension field at bytes[4..7]) requires either (a) threading `extendedLength: UInt32?` through `Request.decode` + per-decoder use in ~5 decoders (PolyLine, PolyArc, ChangeProperty, SetClipRectangles, FillPoly), or (b) a request-rewrite trick that caps support at 65535 4-byte units (defeats the purpose). Real implementation is ~200 lines, not 24. The "wedge risk" the comparison study mentioned (client sending length=0) is now closed by the bogus-length-actually-closes fix from 2026-05-14 â€” we emit BadLength + close, which is spec-correct for an unsupported extension. Revisit when a client we host actually needs BIG-REQUESTS (none today). Confirmed via comparison study: only SHAPE and BIG-REQUESTS matter for the R6/CDE/Motif client mix; neither is on a critical path today.

### Multi-client gaps

- **KillClient / SetCloseDownMode accepted as no-ops.** Decoders shipped 2026-05-14 so these stop falling through to BadRequest (Xt scatters them around session lifecycle). Real implementation would route KillClient.resource â†’ owning client â†’ close socket, and SetCloseDownMode would persist the mode on the session for cleanup-time dispatch. No client we host today actually uses RetainPermanent / RetainTemporary close-down.
- **GrabServer / UngrabServer accepted as no-ops.** Honoring would require coordinator-mediated pause of other sessions' request dispatch. - **Cross-session AlreadyGrabbed not detected on GrabPointer / GrabKeyboard.** `pointerGrab` / keyboardGrab are session-local; we don't coordinate via ServerCoordinator. Two sessions can each call GrabPointer and both believe they hold it. Latent â€” no host client today does cross-session pointer grabs.
