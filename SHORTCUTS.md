# Shortcuts and stubs

Per the XError-honesty policy locked into `CLAUDE.md` (Working
conventions) and `DECISIONS.md` (2026-05-13 entry), this file is the
**active ledger of currently-justified lies and partial implementations**,
not a wish list. Every entry should have:

- A specific file or component reference
- What's stubbed / hardcoded
- Why it's load-bearing (or what client behavior is at stake)
- What "real" would look like

When an entry is converted to real behavior, move it to **Closed** with
a date and rationale.

## Convention

- **When you hardcode** something to make a bigger thing work, add an
  entry here in the same change.
- **When you replace** a hardcode with a real implementation, move the
  entry to the Closed section with a date.
- Periodically review the open list. If it's growing and not shrinking,
  that's a signal to stop adding features and clean up.

## Open

### Resources, tables, replies

- **SetupAccepted is hardcoded.** Lives in `Sources/SwiftXServerCore/ServerConfig.swift` as `ServerConfig.default`. One screen 1280×1024, one PseudoColor 8-bit visual (id 0x22), one pixmap format (depth=8 bpp=8 pad=32), vendor "swift-x", root=0x28, defaultColormap=0x21, resourceIdBase=0x4400000. The matched-Sun values are used so unit tests can replay the captured Sun C2S byte stream against our server. Real version: derive from actual macOS display geometry, expose both PseudoColor 8-bit and TrueColor 24-bit per `DECISIONS.md`.
- **RESOURCE_MANAGER is a 7-line hand-curated Tier 1 fixture.** `Sources/SwiftXServerCore/DefaultMotifResources.swift` publishes a small `*XmText.fontList` / `*XmLabel.fontList` / etc. set targeting `-adobe-helvetica-*` and `-adobe-courier-*` XLFDs (per MOTIF_TEXT_QUALITY.md Tier 1). Steers Motif to pick fonts that route cleanly through `FontResolver`'s substitution table to Mac fonts that render nicely at retina. This is NOT a lie on the wire; documented here so the staged delivery (Tier 2 config file, Tier 3 settings UI) has a home. Real version: full xrdb-equivalent that loads and merges user/system Xresources, exposed editably to the user.
- **Root-window properties are per-session, should be server-global.** `ServerSession.properties` is a `PropertyTable` instance owned by the session, not the coordinator. Properties set on the root window by one client (e.g., a property like `TT_SESSION` that ttsession sets via `ChangeProperty(root)` to advertise its RPC address) live only as long as that client's session — once the client disconnects, the property goes with it, and a subsequent session sees a fresh empty root. Per X11 spec, the root window is server-global; properties on it are owned by the server and outlive any individual client's connection. Concrete instance found 2026-05-18: ttsession on u5 connected to swiftx, did `GrabServer + ChangeProperty TT_SESSION + UngrabServer + disconnect` in a 29-request session, but when dtpad's main session later read the root window, `TT_SESSION` was gone. This is a real architectural bug independent of CDE support — any X11 deployment with WMs / session managers / inter-client property publishers will be affected. Real version: move `PropertyTable` (or at minimum the root-window subset of it) to `ServerCoordinator` so it's shared across sessions, with proper PropertyNotify event routing to all sessions that have PropertyChangeMask on the root.
- **AllocColor returns synthetic monotonic pixel values, per-session.** `Sources/SwiftXServerCore/ColorTable.swift` allocates from `nextPixel = 16` and caches `pixel → RGB` for draw-time lookup. ColorTable is owned by the session, not the coordinator, so two clients can each call AllocColor and receive the same pixel ID with different RGB values; whichever client draws last to a given pixel wins on screen. Real version: PseudoColor colormap with shared cells, freelist, StoreColors, and coordinator-shared state (or per-colormap state on a future Colormap resource type).
- **Color resolution falls back to black for unknown pixels.** `ServerSession.resolveColor` returns black when `ColorTable.rgb(for:)` is nil. Replayed captures hit this constantly because the captured CreateGC bytes reference pixels that the *original* Sun server allocated, not ours. For live clients this is fine because they see our AllocColor reply and use our pixel values.
- **Synthesized xlsfonts is the Phase 1 set, not the full ~100.** `SynthesizedFonts.phase1Names` ships ~40 entries (cell aliases + substitute families × 5 sizes × medium/bold roman iso10646-1). Phase 4 expands to italic + iso8859-1 cross product.

### XError emission (partial)

- **XError emission incomplete on a handful of paths.** Framer's `XError.encode(...)` + `ServerSession.emitError(...)` infrastructure (2026-05-14) and ~55 handlers have been converted across the validation sweep. Still silent-dropping or faking: (a) decode failures within known opcodes — the framer throws but ServerSession's `tryRequest` swallows without emitting BadLength / BadValue; (b) RecolorCursor pixel-values not honored (cursor itself validates → BadCursor); (c) PutImage pixel data dropped (drawable + GC validated → BadDrawable / BadGC; pixel data silent-drop is a ledgered lie tied to "no pixmap pixel storage"); (d) WarpPointer doesn't move the macOS pointer (separate entry below); (e) SendEvent propagation not honored (separate entry below). Captured-app replay tests tolerate BadAtom / BadWindow on property opcodes (Sun-server-internal IDs we never see CreateWindow / InternAtom for) and BadRequest on baseline-allowed extension opcodes.

### Drawing + pixmaps

- **PutImage validates args but doesn't decode pixel data.** `case .putImage` validates drawable + GC at entry, then silent-drops the pixel data. xeyes/xclock/dtcalc-icon all use it for 1-bit-or-8-bit icon bitmaps; without decode those glyphs don't show. Real version: depth/format-aware decoder (Bitmap d=1, ZPixmap d=1/8 cover all observed traffic per the post-Stage-0 captures) writing into the pixmap's PixelBuffer.context. The draws-into-pixmap infrastructure landed in Stage 1b (2026-05-17) — PutImage is what remains. Targets Stage 3 of PLAN_PIXMAP_AND_COPYAREA.
- **CopyArea: same-window memmove fast path doesn't honor GC clip.** All 5 spec-supported variants ship as of Stage 2 (2026-05-17): window→window same-NSWindow uses bitmap memmove (xterm scroll fast path); window→window cross-NSWindow, pixmap→window, window→pixmap, and pixmap→pixmap all snapshot src as CGImage, crop in image-pixel coords, blit via `CGContext.draw(image:in:)` into dst via `withDrawContext` — clip applied for free on those paths. Only the same-window memmove can't easily scissor per row without becoming much slower; xterm doesn't set clip there, but we log when a client does so future divergence is visible. Defensive depth-match BadMatch not implemented yet (window depth=0 = CopyFromParent sentinel needs resolution before comparing against pixmap depths). Motif and Athena always create pixmaps at root-depth so depth mismatches don't fire in practice for the dt-app / xterm flow we host.
- **NoExpose always emitted after CopyArea.** We don't track GC `graphics-exposures`, so we send NoExpose unconditionally. Spec says only emit if graphics-exposures=True; defaults to True so most clients are fine. A client that explicitly sets it False and ignores NoExpose would see spurious events.

### Window lifecycle

- **Synthesized ReparentNotify uses a constant fake parent ID.** `MockWindowBridge.syntheticParentId = 0xC0FFEE00`. xclock and most apps don't care what this value is, just that the event arrives. If a client tries to manipulate the synthesized parent, they'll get an unknown-window response.
- **NSWindow opens at fixed (100, 100) on screen.** `CocoaWindowBridge.mapTopLevel` ignores the X CreateWindow geometry's x/y for the screen position. Real WMs honour `WM_NORMAL_HINTS` and various positional hints. For now the X coords just decide window size.
- **CWBackPixmap / CWBorderPixmap silently dropped on CreateWindow + ChangeWindowAttributes.** Bits 0 and 2 of the CW value mask are never consulted. The 2026-05-15 ChangeWindowAttributes sweep added storage for the other 13 bits but missed the two pixmap-source ones. Real version: store as a `BackPixmapKind` enum (`.none`, `.parentRelative`, `.pixmap(id)`); the `.none` and `.parentRelative` paths unblock transparent-overlay-widget and twm-style root-tiling clients without needing the pixmap-tile case (which is gated on broader pixmap pixel-storage exercise). Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` finding #5).
- **CWBorderWidth dropped in ConfigureWindow.** `case .configureWindow` reads x/y/width/height/stackMode/sibling but not borderWidth (`WindowTable.resize` has no borderWidth parameter). ConfigureNotify reports the stale borderWidth. Motif's BulletinBoard widget changes borderWidth on focus; xfontsel's info-pane uses borderWidth changes. Real version: add to the ValueListReader sweep, plumb through resize, repaint border ring on change, use the new value in the emitted ConfigureNotify. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` finding #6).
- **ReparentWindow on a mapped window doesn't emit the Unmap/Map pair.** Spec 10.4 mandates that ReparentWindow on a mapped window does an implicit UnmapWindow, parent change, then implicit MapWindow if previously mapped — emitting UnmapNotify and MapNotify around the ReparentNotify. Our handler emits only ReparentNotify and never toggles `entry.mapped` or calls bridge unmap/map. Any toolkit that re-parents a mapped widget mid-session breaks; virtual-root WMs break. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` finding #8).

### Input + cursors

- **WarpPointer doesn't move the macOS pointer.** Updates `lastPointerXY` so QueryPointer reflects the warp; doesn't call CGWarpMouseCursorPosition (jarring user behavior in rootless). Clients that warp-and-expect-pointer-visibly-jumped see the discrepancy.
- **SendEvent doesn't propagate.** Sets the synthetic-event high bit on byte 0 and writes to the client. Propagation up the ancestor chain (when propagate=true) is NOT honored — delivers to destination as supplied. Predefined destinations (PointerWindow=0, InputFocus=1) are not resolved.
- **Cursor substitution covers ~10 X glyphs**, NSCursor side. CreateGlyphCursor / FreeCursor / CWCursor in CreateWindow + ChangeWindowAttributes shipped 2026-05-09. NSCursor mapping in `CocoaWindowBridge.nsCursor(forXCursorGlyph:)` covers XC_xterm/iBeam, XC_left_ptr/arrow, XC_crosshair, hand cursors, fleur/openHand, and the resize variants. Unknown glyphs fall back to `.arrow`. **RecolorCursor**: validates cursor → BadCursor on unknown; we don't honor fg/bg colors (NSCursor substitution). **CreateCursor (op 93)**: shipped 2026-05-18 as a stub — pixmap source/mask are validated for existence (BadPixmap) but the bitmap contents are never decoded; cursor renders as NSCursor.arrow at crossing time via the sourceGlyph=0xFFFF sentinel. Surfaced when dtterm's Motif Help menu crashed with BadRequest on opcode 93. Real version: decode the 1-bit source bitmap, apply the mask, build an `NSCursor(image:hotSpot:)`. Custom cursor rendering for Motif resize/busy/menu indicators is the user-visible regression.
- **Keyboard map is US-ASCII only.** `Sources/SwiftXServerCore/USKeymap.swift` covers letters / digits / punctuation / arrows / modifiers via macOS NSEvent virtual keyCode +8 = X keycode. Phase 4: international layouts, dead keys, IME.
- **GetPointerMapping reports [1,2,3]; SetPointerMapping is silently dropped.** macOS scroll-wheel events translate to X buttons 4/5 elsewhere in our event path, but clients that probe `XGetPointerMapping` to decide whether to handle button 4/5 (xterm `-mc on` mode, dt's wheel-scroll, modern plotting clients) skip wheel handling because the map says "3-button mouse." Also: a left-handed user can't swap LMB/RMB because SetPointerMapping doesn't persist. Real version: report [1..5] and persist a mutable per-session mapping. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_replies.md` finding #10).
- **Cross-NSWindow drag tracking via NSEvent.addLocalMonitorForEvents.** XQuartz uses macOS-private `xp_*` kernel APIs (`xp_lookup_native_window`, `xp_window_id`) to inject pointer events at a layer below NSEvent — see `reference/xquartz-xserver/hw/xquartz/X11Application.m:246-480`. We don't have those APIs from a regular Swift AppKit app. Workaround: when an X-protocol pointer grab activates (passive grab, implicit grab on first ButtonPress, or explicit GrabPointer), `CocoaWindowBridge.startCrossWindowDragTracking` installs a local NSEvent monitor for `[.*MouseDragged, .*MouseUp]` so we can intercept events before AppKit's responder chain. The monitor computes the global pointer position, looks up which managed NSWindow contains it (popup-level NSPanels first per z-order), translates to that window's logical X coords, and routes via `fireMouseDragged` / `fireMouse`. The monitor consumes the event so the origin's FlippedXView doesn't double-fire. Released when the grab ends. Caveat: AppKit's `event.deltaX/deltaY` is unreliable after Spaces/Fast-User-Switching per X11Application.m:929-936; we use `locationInWindow + window.convertPoint(toScreen:)` which sidesteps that issue.

### Selection / clipboard

- **Selection roundtrip only handles STRING, not INCR.** `case .changeProperty` intercepts converted text on `selectionSinkWindow` and pushes to NSPasteboard, decoding bytes as ISO-8859-1. Big selections (xterm uses INCR > 64K) and non-STRING targets (UTF8_STRING, COMPOUND_TEXT) aren't handled. Typical xterm selections are short ASCII so this works in practice; flag if multi-MB selections start truncating.
- **CLIPBOARD selection isn't mirrored.** Only PRIMARY (atom 1) is tracked for the copy roundtrip. xterm-on-u5 only writes PRIMARY so this is fine; modern xterm builds and Motif apps that write CLIPBOARD won't end up on NSPasteboard until we add it.
- **SetSelectionOwner has no time-comparison gate.** Spec 4.2.1: silently ignore when `r.time < prior.time` or `r.time > serverTime`. `ServerCoordinator.swapSelectionOwner` unconditionally overwrites without consulting the prior owner's stored time. Latent because today's hosted clients don't contend; the day two clients race for PRIMARY, a late-arrival ConvertSelection can clobber a newer owner. ~5 line fix. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_replies.md` finding #7).
- **GetProperty doesn't honor the type filter.** Spec: when the request specifies a non-zero type and the stored property's type differs, the reply returns the stored property's actual type + bytes-after=full-length + an empty value (don't transfer bytes). We never compare stored type against requested type and always transfer. This is the bug-shape the moment anyone implements INCR — the INCR pattern reads with delete=True on every intermediate chunk and gates on type match. Also: GetProperty(delete=True) currently deletes regardless of `bytes-after`; spec says delete only when bytes-after==0. Surfaced by comparison fork 2026-05-14 (`archive/comparison/risk_selections_properties.md`).

### Known visual / behavioral residue

- **dt-Motif widget chrome doesn't redraw on Expose — four hypotheses now closed in code; pending u5 verification.** dt-apps render their containers + LCD area + any window with explicit non-bg `BackPixel`, but the deep button hierarchy historically rendered as flat panels with no visible button shadows or labels. Per gold-vs-swiftx trace diff 2026-05-10: gold emits ~7 Expose events for the whole calculator boot, we emitted 451. Region Steps E0+E1+E1.5+E2 (2026-05-13) collapsed dtcalc Expose count to 8, matching gold within 1. SubstructureNotify variants of MapNotify (and friends) shipped 2026-05-14. VisibilityNotify state-transition emission shipped 2026-05-14. **Fourth fix 2026-05-14 (post-comparison-study):** visibility state was derived from `clipList` (post-children-subtraction) but spec says compute it IGNORING subwindows. Container windows fully covered by their child widgets were therefore reported as FullyObscured, which is exactly the signal Motif's PushButton uses to skip shadow-chrome drawing on Expose. `emitVisibilityChanges` now derives state from `borderClip ∩ interiorBox`; new test `VisibilityNotifyTests.testContainerCoveredByChildrenStaysUnobscured` locks it in. Surface u5 next session — if chrome still doesn't appear, hypothesis pool is empty and we go deeper on event ORDERING.
- **Popup menu placement is over the launching button instead of below it.** Visible in both xfontsel (Athena MenuButton + SimpleMenu) and quickplot (Motif menu). Override-redirect popup window is positioned where the client requested, but Motif/Athena typically request a position that assumes the menu drops downward FROM the bottom edge of the launching button; our placement appears to land directly on the button. Possible causes: we don't honor `_MOTIF_WM_OFFSET` on the popup, or our coordinate origin for override-redirect windows is off by the height of the launching widget. Cosmetic but consistent across toolkits.
- **Motif Text widget character spacing.** Plot text + xcalc PolyText8 render correctly so global font metrics are fine; Motif's XmText widget specifically shows odd inter-glyph spacing. Likely Motif treats Text as column-based and we're returning Core Text actual-advance values that don't snap to its column model. Inspect when revisiting.
- **Idle 6-request poll loop in quickplot.** When quickplot sits idle, the server log streams a tight repeat of `GetWindowAttributes, GetGeometry, ChangeGC, PolySegment, ChangeGC, PolySegment` — top-shadow + bottom-shadow stroke pair on something that introspects window state each pass. Probably a Motif Text widget cursor blink or focus-ring animation timer that never decides nothing has changed. Wastes server CPU but renders correctly. Compare to gold capture before claiming it's our bug — Motif may behave this way against any X server.
- **No SHAPE extension.** XShape is what gives xeyes its non-rectangular outline (eye shape clipped to an ellipse). Without it, xeyes renders inside its rectangular bounding window — visible as a slight rectangular "halo" around each eye where the white sclera meets the window edge. Cosmetic, low-priority. xclock has the same issue for its non-square clock face.
- **Expose-vs-ConfigureNotify cascade divergence during early Motif widget packing.** Compared to dtcalc-running-on-u5 gold (which shows a clean `ConfigureNotify 143x{46,67,69,90,111,134,...}` cascade as Motif's container manager packs each row), our session emits mostly Expose events on the same windows plus a single ConfigureNotify. Likely a downstream symptom of font metrics divergence (Motif's packing math returns different ConfigureWindow calls when widget sizes are reported differently); could also be a missing ConfigureNotify on a geometry change we should be emitting on. Verify after the QueryFont substitution path settles. Possibly the upstream cause of the "xcalc resize moves text but not buttons" symptom. Surfaced in the dtcalc-sun vs dtcalc-swiftx diff 2026-05-17 (preserved findings from the archived FOLLOWUPS doc, item #2).

### Extensions

- **BIG-REQUESTS not advertised.** DECISIONS 2026-05-05 committed to ship; deliberately re-deferred 2026-05-14 after investigating cost. R6's server-side dispatch handler IS small (~24 lines per `Xext/bigreq.c`), but R6's decoders read body lengths from a `client->req_len` field in a context struct, not from header bytes. swift-x's `Request.decode` and every per-opcode decoder reads `lenIn4` from `bytes[2..3]`, so supporting extended-length encodings (length=0 → 32-bit extension field at bytes[4..7]) requires either (a) threading `extendedLength: UInt32?` through `Request.decode` + per-decoder use in ~5 decoders (PolyLine, PolyArc, ChangeProperty, SetClipRectangles, FillPoly), or (b) a request-rewrite trick that caps support at 65535 4-byte units (defeats the purpose). Real implementation is ~200 lines, not 24. The "wedge risk" the comparison study mentioned (client sending length=0) is now closed by the bogus-length-actually-closes fix from 2026-05-14 — we emit BadLength + close, which is spec-correct for an unsupported extension. Revisit when a client we host actually needs BIG-REQUESTS (none today). Confirmed via comparison study: only SHAPE and BIG-REQUESTS matter for the R6/CDE/Motif client mix; neither is on a critical path today.
- **SolarisIA extension missing.** Real Sun's Xsun advertises it (`QueryExtension(SolarisIA) → present=true major=135`); we report present=false. SolarisIA is the Solaris-internal Inter-client Atoms cache. When absent, Motif falls back to interning ~50 atoms one-by-one at startup. No visual bug — just latency and log noise during dt-app boot. Real fix: minimal extension implementation that batches atom interning (~half day). Surfaced in the dtcalc-sun vs dtcalc-swiftx diff 2026-05-17 (preserved findings from the archived FOLLOWUPS doc, item #3).
- **RotateProperties (op 114) not wired.** No framer decoder; falls through to BadRequest. Used by Xt for cycling icon-name properties and by some twm-era utilities. Real version: decoder + handler that rotates a list of N atoms' values by `delta` positions and emits PropertyNotify(NewValue) per atom (~50 lines). Surfaced by audit fork 2026-05-15 (`archive/audit/audit_replies.md` finding #11).
- **Input/keyboard opcodes still falling through to BadRequest.** ChangeKeyboardMapping (100), ChangeKeyboardControl (102), GetKeyboardControl (103), ChangePointerControl (105), GetPointerControl (106), SetPointerMapping (116), SetModifierMapping (118). Spec-defined opcodes that Motif/Xt may probe at startup; no client we host today breaks visibly, but they generate misleading "server is broken" log entries in Xt error handlers. Real fix: add framer decoders, route each to either a spec-correct no-op-or-stored-value response or its spec-mandated error code (`BadValue` / `BadMatch` per opcode). Surfaced by comparison fork 2026-05-14 (`archive/comparison/risk_input.md` + `risk_errors.md`).

### Multi-client gaps

- **KillClient / SetCloseDownMode accepted as no-ops.** Decoders shipped 2026-05-14 so these stop falling through to BadRequest (Xt scatters them around session lifecycle). Real implementation would route KillClient.resource → owning client → close socket, and SetCloseDownMode would persist the mode on the session for cleanup-time dispatch. No client we host today actually uses RetainPermanent / RetainTemporary close-down.
- **GrabServer / UngrabServer accepted as no-ops.** Honoring would require coordinator-mediated pause of other sessions' request dispatch. WMs that use GrabServer for atomic multi-request changes won't get atomicity against other clients. No client we host today exercises this. See `OPCODE_STATUS.md` row 36.
- **Cross-session AlreadyGrabbed not detected on GrabPointer / GrabKeyboard.** `pointerGrab` / keyboardGrab are session-local; we don't coordinate via ServerCoordinator. Two sessions can each call GrabPointer and both believe they hold it. Latent — no host client today does cross-session pointer grabs.
- **Substructure-redirect events (MapRequest / ConfigureRequest / CirculateRequest) never emitted.** Anywhere the spec says "if parent has SubstructureRedirectMask, redirect to the manager," we apply the request unconditionally — the *Request events aren't produced anywhere in the codebase. Invisible single-client today; blocks any "swift-x is the inner server under an outer WM client" scenario. Worth flagging in DECISIONS / non-goals if we're not planning to host external WMs. Surfaced by audit fork 2026-05-15 (`archive/audit/audit_state_mutation.md` pattern D).

### Operational

- **Server appears in the Dock.** `setActivationPolicy(.regular)` is what gives us the menu bar; the side effect is a Dock icon. Suppressing it needs an `Info.plist` with `LSUIElement=1` (or `NSUIElement=YES`), which Swift Package executables don't ship by default. Either start producing an .app bundle with a real Info.plist, or call the private `TransformProcessType` runtime trick. Cosmetic, deferred.
- **Listening IP detection picks the first `en*` interface.** `primaryLocalIPv4()` in `main.swift` returns en0 if up, otherwise the first up `en*` interface; skips `awdl*`, `utun*`, `bridge*`, etc. Hosts with multiple LAN interfaces (Mac Pro with both Wi-Fi and Ethernet on different subnets) may show the wrong one. Workaround today is `--host <ip>` to override.
- **No client authentication.** TCP listener binds `0.0.0.0:6000`; auth-protocol-name and auth-protocol-data fields on SetupRequest are parsed off the wire and discarded. `SetupRefused` is plumbed in the framer but never produced. Acceptable for the LAN-only vintage-Sun use case; not acceptable once anyone else might run this. Real fix: MIT-MAGIC-COOKIE-1 implementation plus a Unix-socket bind option. Surfaced by comparison fork 2026-05-14 (`archive/comparison/risk_connection_setup.md`).
- **motionBufferSize advertised as 256 in SetupAccepted, GetMotionEvents always returns 0 events.** `ServerConfig.swift` advertises a non-zero ring; the handler at `ServerSession.swift` returns nEvents=0 because there's no ring. Lie-on-the-wire that no client we host today probes. Cheap honest fix: drop the advertise to 0. Real fix: sample pointer motion into a ring buffer (~50 lines). Surfaced by audit fork 2026-05-15 (`archive/audit/audit_replies.md` finding #9).
- **CDE atoms not pre-interned at server startup.** Clients see different atom IDs vs Sun's pre-allocated set; clients work fine via InternAtom roundtrip but the diff against u5 gold has ~50 lines of atom-intern noise that aren't real divergence. Cheap fix: pre-intern the standard CDE/Motif atom set in `AtomTable.init()` (the ~50 atoms SolarisIA would otherwise cache plus the `_MOTIF_*` family). Buys quieter logs and better capture-diff signal. Surfaced in the dtcalc-sun vs dtcalc-swiftx diff 2026-05-17 (preserved findings from the archived FOLLOWUPS doc, item #4).

## Closed

- **Fake CDE customization daemon + hardcoded SDT Pixel Set + CDE-flavored RESOURCE_MANAGER fixture all retired.** Closed 2026-05-18 (see DECISIONS.md 2026-05-18 entry). The 2026-05-10 wedge diagnosis that drove the impersonation was wrong — the real wedge was our own `MATCH_SELECT`-time bug, fixed separately. SS2 gold capture shows SS2 publishes none of these. With the fixes commented out in `ServerSession.swift`, dtcalc + dtterm + quickplot render with full SS2 visual parity (Motif fallback blue). The `ColorTable.swift` pre-seeded CDE palette (pixels 1-23) is still in code but dormant — no SDT Pixel Set indirection routes through it anymore; deletion deferred pending a few more dt-app smoke runs.
- **Descendant geometry change leaves stale pixels on parent.** Fixed via Region Step E1.5 (2026-05-13). `ConfigureWindow` on a non-top-level captures the pre-move `borderClip`, computes `oldBorderClip - newBorderClip` after the recompute, and via `ServerSession.repaintParentOverUncovered` paints the parent's bg over the delta (top-level coords) and emits Expose to parent for that region (parent-local coords) if it has ExposureMask. xcalc Athena re-layout after resize should no longer ghost buttons at their pre-resize positions. Unverified visually until u5 live test.
- **EnterNotify / LeaveNotify with the standard X11 detail-field rules.** Shipped 2026-05-09. Tracking area on FlippedXView delivers `mouseMoved` / `mouseEntered` / `mouseExited`; bridge plumbs to ServerSession's `handlePointer*` methods which compute the crossing chain (Ancestor / Inferior / Nonlinear / Virtual / NonlinearVirtual) and emit only to windows whose eventMask has EnterWindowMask or LeaveWindowMask. (Active-grab crossing modes NotifyGrab / NotifyUngrab shipped alongside GrabPointer / GrabKeyboard.)
- **Expose flood on dt-Motif map.** Closed via Region Steps E0/E1/E1.5/E2 (2026-05-13). MockWindowBridge.emitMapSequence + recomputeClipsForSubtreeContaining now emit Expose per-clipList-rect in window-local coords with proper `count` field; fully-covered descendants emit zero Exposes. dtcalc Expose count went from 248 → 8, matching gold within 1.
- **Reply-generating opcodes that previously silent-dropped.** The 2026-05-10 sweep landed QueryTree, GetAtomName, TranslateCoordinates, QueryPointer, ListExtensions, QueryKeymap as proper replies; QueryBestSize landed earlier. None are silent-dropped — see OPCODE_STATUS.
- **GC clip rectangles honored for poly/image-text draws.** Shipped 2026-05-14. `WindowBridge` draw methods take `clipRectangles: [Rectangle]?`; `CocoaWindowBridge.withClip` applies the union via `CGContext.clip(to:)` inside saveGState/restoreGState. Still NOT honored on CopyArea (memmove path doesn't go through CGContext) — tracked in the cross-window CopyArea entry above.
- **GC dashes honored on stroke methods.** Shipped 2026-05-14. `WindowBridge` stroke methods (drawPolySegment / drawPolyLine / drawPolyRectangle / drawPolyArc) take `dashes: [UInt8]?` + `dashOffset: UInt32`; `CocoaWindowBridge.applyDashes` calls `CGContext.setLineDash`. Single-byte patterns expand to [N, N] per X spec.
- **GrabKey passive grabs honored.** Shipped 2026-05-14. `findActivatablePassiveKeyGrab` mirrors the button-grab match algorithm; `handleKeyEvent` routing precedence is now: active keyboard grab → matching passive key grab → focus → keyTarget fallback. Should make quickplot accelerators fire; visual verification pending u5.
- **SubstructureNotify chains** across Create / Map / Unmap / Destroy / Reparent / Configure + their *Subwindows variants. Shipped 2026-05-14. `ServerSession.notifySubstructure(parent:build:)` helper: if `parent` has SubstructureNotifyMask, builds and appends the event with `event = parent`. UnmapWindow / UnmapSubwindows additionally repaint the parent's bg over the unmapped child's previously-covered region + emit Expose to parent if ExposureMask is set.
