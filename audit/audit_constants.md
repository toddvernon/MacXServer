# Cross-cutting hardcoded-constants audit

Pass across `Sources/SwiftXServerCore/**` + `Sources/Framer/Replies/**`,
asking the QueryTextExtents question of every literal: "is this value
right because of physics, or right because the corpus happens to
have it that way?"

Skipped: reply markers (the leading `1` byte on every reply), the
unused-byte padding, opcode constants (`62` for CopyArea major in
`NoExposureEvent`, etc.), enum rawValues. Those are obviously right.

## Pattern I noticed

The accidental corpus-matches cluster in **two zones**, both related to "describe the font / the colormap":

1. **Font metrics replies** (`QueryFontReply`, `QueryTextExtentsReply`): we picked the *monospace* shape (min=max, charInfos empty, defaultChar=space, allCharsExist=true, range 32..126, overallLeft=0) and ship it for every font. This is the QueryTextExtents bug *frozen into a helper function* that hasn't yet been called with a proportional font during a test that actually checks the reply contents. The replay tests passed because Monaco is monospace and the corpus didn't include any italic font measurements.

2. **The CDE palette pre-seed in `ColorTable.swift`** is the most dangerous unflagged shortcut in this audit. RGB values are reverse-engineered from a specific dtcalc capture (`GC 0x440000A: FG=0x9 BG=0xD` etc. in the comments). It works for dt-apps because it was built from dt-apps. A non-CDE client that AllocColor'd pixel 0x09 on a real Sun expecting the dtsession-installed value gets *our guess at CDE Default scheme*, not the actual color. This isn't a fix-by-shortcut, it's a fix-by-corpus-fingerprinting.

The handshake-time constants in `ServerConfig.swift` (resource IDs, vendor, release) are corpus-matches by design and harmless. The dangerous matches are the ones inside *reply handlers* that compute a value per-request — those are where "right for the corpus, wrong for a new client" lives.
